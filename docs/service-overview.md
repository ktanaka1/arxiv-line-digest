# サービス概要

## サービス名
arxiv-line-digest（仮）

## 何を作るのか
arXiv の新着・話題の論文を自分の興味に合わせて自動で絞り込み、既存の LINE 動線に1日数本だけプッシュ通知する個人用bot。

## ターゲットユーザー
作者本人のみ。フォロワー獲得・対外発信は目的にしない（読むのは自分だけ）。

## 大切にしたいこと
- **通知過多にしない**：1日数本だけ。母数（3カテゴリで1日数百本）を絶対に全部流さない。
- **実用性で絞る**：「明日からの業務・個人開発ですぐコードに落とし込めるか」を判定軸にする。
- **動線を増やさない（方針）／実態は専用チャンネル**：当初は既存プロジェクトの LINE 動線に相乗りする
  方針だったが、Stock Signal の通知と論文通知が混在して区別しづらいため、専用の LINE 公式アカウント
  「ArXiv Digest」を新規作成した（プロバイダー「AI活用Pjt」配下）。1チャンネルで完結する点は維持。
- **狭く始める**：対象カテゴリ・機能は最初は絞り、物足りなければ後で足す。

## ビジョン（最終目標）
「毎日arXivを手で追うのが面倒」を解消し、スマホのプッシュ通知で1日数本だけ、自分に刺さる論文が届く状態を作る。

## MVP（最低限必要な機能）
1. **arXiv新着の取得**：`cs.AI` `cs.LG` `cs.CL` の新着を取得する。
2. **系統A 🔵🆕 新着系**：新着 →（キーワード一次フィルタ）→ LLM採点（興味ルーブリック）→ 高評価のみ1日数本。
3. **系統B 🟡🌟 生き残り系**：数週間スケールで伸びた論文を通知。シグナル＝GitHubスター急伸（Papers With Code で arXiv→repo 紐付け → GitHub API）/ Hugging Face トレンド / 良質サーベイ。
4. **LINE通知**：既存 LINE Messaging API 動線へ、系統を色・絵文字で識別できる文面で配信。
5. **重複排除**：通知済み論文を記録し、再通知を防ぐ（系統間の再浮上含む）。
6. **定期実行**：1日1回、朝に実行（arXiv の更新が平日 0:00 UTC 頃に1日1回のため、朝1回で十分）。

## 興味プロファイル（系統A LLM採点の基準）
- 人物像：実装・ハッカー気質（Pragmatic）。
- 好むテーマ：RAGの高度化 / AIエージェント / プロンプトエンジニアリング / API連携 / OSSモデルのファインチューニング。
- 加点：手を動かせる（コード/リポジトリ/レシピが具体的）、テーマ合致、個人〜小規模で再現可能、すぐ効く実務Tips。
- 減点・除外：理論・証明オンリー、ベンチSOTAの僅差更新、大手しか再現できない、応用の利かないニッチ特化。
- 方針：「読んだ実務者が明日コードを書きたくなるか」を10点満点で採点し、閾値以上のみ通知。

## 将来的に追加したい機能
- 対象カテゴリの追加（物足りなければ）。
- 実行頻度の最適化（朝夕2回など）。
- 通知文面のリッチ化（1行要約・スコア表示など）。

## 不採用・見送り（意図的な判断）
- **Bluesky 連携**：Blueskyの通知は「他者からの反応」が対象で、特定アカウントの新規投稿を毎回プッシュする用途に合わないため不採用。
- **被引用数の急伸シグナル**：引用は反応が半年〜1年と遅く、数週間スケールでは動かないため系統Bに採用しない。
