# arXiv論文 自動通知bot — 仕様化ドラフト

> 壁打ち（my-project）からの書き出し。仕様化フォルダにコピーして本仕様に育てる用のドラフト。
> 作成日: 2026-06-17

## 1. 目的・ゴール

- **自分用の論文情報収集を自動化する**ためのbot。
- フォロワー獲得・対外発信は目的にしない（読むのは自分だけ）。
- 「毎日arXivを手で追うのが面倒」を解消し、**スマホのプッシュ通知で1日数本だけ**届く状態を作る。

## 2. 通知先（配信動線）

- **既存の別プロジェクトの LINE Messaging API 動線に相乗りする。**
- 通知チャンネルは **1つ**。アカウントを増やさない。
- 1つのトーク画面に、後述の2系統が**色・絵文字で識別できる形**で混在して流れる。
  - 系統A: 🔵 ブルー / 🆕
  - 系統B: 🟡 ゴールド / 🌟
- ※当初Bluesky APIを検討したが、Blueskyの通知は「他者からの反応（メンション等）」が対象で
  「特定アカウントの新規投稿を毎回プッシュ」が標準でないため不採用。既存LINE動線に一本化。

## 3. 対象ソース・分野

- **arXiv** の以下カテゴリのみに絞る（最初は狭く始め、物足りなければ後で足す）:
  - `cs.AI`（人工知能）
  - `cs.LG`（機械学習）
  - `cs.CL`（自然言語処理）
- 母数の目安: 上記3カテゴリで **1日あたり数百本**（全分野だと1日1万本規模 → 絶対に全部は流さない）。

## 4. 2系統の通知ロジック

1チャンネルに2種類のロジックの通知を混在させる。

### 系統A 🔵🆕 新着系
- **今日のarXiv新着**から、LLMが「ユーザーの興味に合う」と判定した論文を通知。
- 絞り込みパイプライン: 新着数百本 →（キーワード一次フィルタ）→ LLM採点 → **1日数本**。

### 系統B 🟡🌟 生き残り系
- 過去数週間で「**生き残った／効いている**」論文を通知。
- **シグナル（数週間スケールで実際に動くもの）:**
  - ✅ **GitHubスター急伸**（Papers With CodeでarXiv→repo紐付け → GitHub APIでスター推移）
  - ✅ **Hugging Face トレンド**（HF Papers / Daily Papers）
  - ✅ **良質なサーベイ論文**（タイトルの "survey"/"review" 等 + LLM判定）
  - ❌ **被引用数の急伸は不採用**（引用は反応が半年〜1年と遅く、数週間では動かないため）

## 5. 興味プロファイル（LLM採点の基準）

- **人物像: 実装・ハッカー気質（Pragmatic）。**
- **好むテーマ:** RAGの高度化 / AIエージェント / プロンプトエンジニアリング / API連携 / OSSモデルのファインチューニング。
- **刺さるポイント:** 「明日からの業務や個人開発ですぐコードに落とし込んで試せそう！」という実用性。

### 系統A LLM採点ルーブリック（たたき台）
**加点（通知する）**
- ✅ 手を動かせる（コード/リポジトリ/プロンプト/レシピが具体的）
- ✅ テーマ合致（RAG・エージェント・プロンプト技法・API連携・OSS FT）
- ✅ 再現可能なスケール（個人〜小規模で試せる。ローカルGPUやAPI呼び出しで完結）
- ✅ すぐ効く知見（「この一工夫で精度/コスト改善」など実務Tips）

**減点・除外（弾く）**
- ❌ 理論・証明オンリーで実装イメージが湧かない
- ❌ ベンチマークSOTAを0.x%更新しただけのマイナー改良
- ❌ 巨大事前学習など大手しか再現できないもの
- ❌ 特定ニッチタスク特化で応用が利かない

**採点プロンプト方針:** 「この論文を読んだ実務者が、明日コードを書きたくなるかを10点満点で採点。7点以上のみ通知」程度の指示でユーザー感覚に寄せる。

## 6. 仕様化フェーズで詰める残論点（未決定）

- 実行頻度: 1日1回 / 朝夕2回 など。
- キーワード一次フィルタを噛ませるか（コスト削減 vs 取りこぼし）。
- LLM採点の閾値（7点以上、など）の具体値とモデル選定。
- 通知メッセージの文面フォーマット（タイトル / 1行要約 / リンク / スコア表示の有無）。
- 系統Bの「急伸」判定の具体しきい値（スター増加率、期間）。
- 1日の通知上限件数のキャップ。
- 重複排除（系統Aで通知済みの論文が系統Bで再浮上したときの扱い）。
